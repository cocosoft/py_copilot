# 知识库处理流程

## 标准处理流程

```
┌─────────────────┐
│  1. 文档预处理   │
│  (清洗、分块)    │
└────────┬────────┘
         ▼
┌─────────────────┐
│  2. 实体识别     │
│  (NER/关系抽取)  │
└────────┬────────┘
         ▼
┌─────────────────┐
│  3. 文档向量化   │
│  (Embedding)    │
└────────┬────────┘
         ▼
┌─────────────────┐
│  4. 知识图谱构建 │
│  (图数据库存储)  │
└────────┬────────┘
         ▼
┌─────────────────┐
│  5. 知识图谱显示 │
│  (可视化展示)    │
└─────────────────┘
```

## 各阶段详细说明

### 1. 文档预处理

- 文档格式转换（PDF/Word/Excel等 → 纯文本）
- 文本清洗（去除噪声、标准化）
- 文档分块（按段落、语义或固定长度切分）

### 2. 实体识别（Entity Recognition）

- 使用 NER 模型识别文本中的**命名实体**（人名、地名、组织、时间等）
- 关系抽取：识别实体之间的关系
- 这是构建知识图谱的基础数据来源

### 3. 文档向量化（Embedding）

- 将文本转换为高维向量（如使用 BERT、OpenAI Embedding 等）
- 用于语义搜索、相似度计算
- 存储到向量数据库（如 Milvus、Pinecone、Chroma 等）

### 4. 知识图谱构建

- 将识别出的实体和关系导入图数据库（如 Neo4j、JanusGraph）
- 建立实体节点和关系边
- 与向量数据建立关联

### 5. 知识图谱显示

- 前端可视化展示（如使用 D3.js、ECharts、Cytoscape.js）
- 支持交互式探索、查询

## 为什么是这个顺序？

| 顺序 | 原因 |
|------|------|
| 实体识别在向量化之前 | 实体识别需要原始文本，且其结果可用于丰富向量元数据 |
| 向量化在图谱构建之前 | 向量用于语义检索，图谱用于结构化查询，两者可并行构建 |
| 显示在最后 | 需要先有数据才能可视化 |

## 实际应用中的变体

### 变体1: 并行处理

```
文档 → [实体识别] ─┬─→ 知识图谱
         ↓         │
      [向量化] ────┘
         ↓
    向量数据库
```

### 变体2: 检索增强生成(RAG)

```
查询向量化 → 向量检索(召回相关文档) → 实体识别(精化答案) → 显示
```

## 技术栈参考

| 阶段 | 常用技术/工具 |
|------|--------------|
| 文档预处理 | Python (PyPDF2、python-docx)、Apache Tika |
| 实体识别 | spaCy、HanLP、BERT-NER、Stanford NER |
| 向量化 | OpenAI API、Sentence-Transformers、BERT |
| 向量数据库 | Milvus、Pinecone、Chroma、FAISS |
| 图数据库 | Neo4j、JanusGraph、NebulaGraph |
| 可视化 | D3.js、ECharts、Cytoscape.js、G6 |
