# 知识库项目结构整理方案

## 一、现状分析

### 1.1 当前问题

通过分析发现项目存在以下结构问题：

#### 问题1：services/knowledge 目录过于臃肿
- **现状**：79个文件全部堆放在一个目录
- **问题**：
  - 难以快速定位文件
  - 职责不清晰
  - 维护困难
  - 示例文件和核心代码混在一起

#### 问题2：命名不规范
- `advanced_text_processor.py` vs `text_processor.py`
- `llm_extractor.py` vs `llm_text_processor.py`
- 多个功能相似的文件并存

#### 问题3：示例文件污染
- `*_example.py` 文件多达10+个
- 与核心代码混在一起

#### 问题4：API层结构混乱
- 知识图谱相关API分散在多个文件
- 缺少统一的知识库管理API

## 六、风险与回滚方案

### 6.1 风险

| 风险 | 等级 | 应对措施 |
|-----|------|---------|
| 导入路径遗漏 | 中 | 使用IDE全局搜索检查 |
| 循环导入 | 中 | 检查__init__.py配置 |
| 功能损坏 | 低 | 完整测试后再删除备份 |

### 6.2 回滚方案

如果出现问题，可以快速回滚：
```bash
# 删除新目录
rm -rf app/services/knowledge

# 恢复备份
mv app/services/knowledge_backup app/services/knowledge
```

## 七、整理后的优势

1. **职责清晰**：每个目录职责明确
2. **易于维护**：文件数量减少，结构清晰
3. **便于扩展**：新增功能有明确位置
4. **代码复用**：通过__init__.py提供统一入口
5. **示例分离**：示例代码不影响核心代码
